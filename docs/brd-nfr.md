# BRD та NFR — TaxiApp

## 1) BRD (Business Requirements Document)

### 1.1. Бізнес-цілі
- Автоматизувати зберігання та доступ до даних сервісу таксі.
- Надати мінімальний набір функцій для пасажира і водія у демо веб.
- Отримувати базові звіти/аналітику по поїздках/тарифах/оцінках.

### 1.2. Ролі користувачів (Actors)
- **Passenger (Пасажир)**
- **Driver (Водій)**

### 1.3. Бізнес-процеси (High-level)
1. Реєстрація/вхід користувача (пасажир/водій).
2. Створення поїздки пасажиром (pickup/destination/tariff).
3. Прийняття поїздки водієм.
4. Зміна статусів поїздки.
5. Завершення поїздки та розрахунок вартості.
6. Залишення відгуку пасажиром.
7. Перегляд активних і минулих поїздок.

---

## 2) Функціональні вимоги (FR)

### 2.1. Аутентифікація та користувачі
**FR-01** Система повинна підтримувати реєстрацію пасажирів.  
**FR-02** Система повинна підтримувати реєстрацію водіїв.  
**FR-03** Система повинна підтримувати авторизацію за email/телефоном та паролем.  
**FR-04** Паролі повинні зберігатися у вигляді хешу.

### 2.2. Тарифи
**FR-05** Система повинна зберігати тарифи (назва, ціни).  
**FR-06** Пасажир при створенні поїздки має обирати тариф.

### 2.3. Поїздки (Rides)
**FR-07** Пасажир може створити поїздку, вказавши адресу подачі та адресу прибуття.  
**FR-08** Система повинна зберігати статус поїздки та дані замовлення.  
**FR-09** Водій може приймати поїздку та оновлювати статуси (accepted/in_progress/completed/canceled).  
**FR-10** Пасажир може переглядати активні та минулі поїздки.  
**FR-11** Водій може переглядати свої активні/виконані поїздки.

### 2.4. Автомобілі
**FR-12** Система зберігає дані авто, прив’язаного до водія.

### 2.5. Відгуки
**FR-13** Після завершення поїздки пасажир може залишити відгук водію (rating + comment).  
**FR-14** Відгук пов’язаний з конкретною поїздкою, водієм і користувачем.

### 2.6. Аналітика (опційно)
**FR-15** Система формує базові звіти (популярність тарифів, середні оцінки, кількість поїздок за період).

## 3) Нефункціональні вимоги (NFR)

### 3.1. Продуктивність
- **NFR-P1:** сторінки демо веб відкриваються без помітних затримок на локальному середовищі.
- **NFR-P2:** запити для історії поїздок оптимізуються індексами.

### 3.2. Надійність і коректність
- **NFR-R1:** БД забезпечує цілісність даних через PK/FK/UNIQUE/CHECK.
- **NFR-R2:** помилки БД коректно обробляються на сервері.

### 3.3. Безпека
- **NFR-S1:** паролі зберігаються як хеш.
- **NFR-S2:** секрети зберігаються у `.env` і не комітяться у репозиторій.

### 3.4. Масштабованість
- **NFR-SC1:** схема БД дозволяє розширення (payments, промокоди, бонуси, історія геолокацій).

### 3.5. Підтримуваність
- **NFR-M1:** документація та README актуальні.
